issue cluster 1 · high severity

Security — The Structural Gap

B-1 No role enforcement at routing layer HIGH

All admin roles share one auth group. Role middleware (CheckRole) is disabled. Any authenticated user reaches any endpoint.

B-1 Dead middleware — re-enabling it would crash the system BLOCKER

CheckRole redirects to route names that don't exist. The natural first fix is itself broken and must be addressed before any auth work begins.

B-5 Student authentication is IC-number only HIGH

No password, no token, no throttling. Anyone who knows a student's IC number has full access to their data.

B-10 Student login page exists — backend support does not HIGH

No student role in middleware. No student records in database seeders. The login flow cannot succeed in any fresh deployment.

06 / 14
issue cluster 2 · high severity

Data Layer — Silent Failures

B-3 MongoDB default + SQL-style migrations HIGH

Every foreign key constraint in every migration file is silently ignored — MongoDB never processes them. The schema documentation describes constraints the database never enforces.

B-2 Achievement schema invariant is broken HIGH

Migration defines a numeric ID column. Controller writes string IDs (PS0001). A foreign key references a column never defined. MongoDB hides all of this silently.

B-4 Payment credentials are hard-coded in a controller HIGH

Secret key, category code, and a dev-environment URL (https://dev.toyyibpay.com) are literal strings in paymentController.php. No environment separation, no adapter interface.

09 / 14
issue cluster 5 · frontend

Frontend — UX and Contract Failures

F-1 Status vocabulary mismatch MED

Backend writes Approved / Permohonan Belum Diproses. Result page modal checks Diluluskan / Dalam Proses — strings the backend never writes. An approved student sees the wrong message and wrong CTA.

F-2 Card layout has overlapping elements MED

Card component elements overlap each other. Fragile layout — breaks on varying content length or screen size.

F-3 Navigation arrows missing from dashboard LOW

Navigation arrows present on landing and login pages, absent from dashboard. UI convention established and then abandoned — components are not being reused across pages.

issue cluster 6 · documentation

Promising More Than Was Built

D-4 Crew application workflow not implemented HIGH

UC009/UC011/UC012 specify: duplicate prevention · rejection reasons · notifications · scheduling · acceptance artifacts. Implementation: a status string and basic views.

D-5 SDD describes an architecture that doesn't exist HIGH

SDD claims role-enforced modular subsystems. Reality: role middleware disabled, one controller for 8–10 subsystems. Anyone reading it to make design decisions is working from a false map.

D-6 SRS and SDD disagree on module count MED

SRS: 4 modules. SDD: claims 4, lists 6. Code: implements 6. No single document is authoritative.

D-2/D-3 Missing use cases + unsupported PWA claim MED

Donation and equipment-report modules have no SRS use cases. PWA described in both documents — no service worker or manifest exists in the repository.

analysis

Why These Issues Exist

1
Framework-led design
Laravel's structure defined the architecture instead of the architecture defining how Laravel was used. Features land in the nearest controller.
2
Authorization by intent, not mechanism
Role boundaries were designed and named, but never operationalized as enforced structural boundaries.
3
Documentation written in isolation
SRS and SDD were produced by separate teams at separate times and never reconciled against the final code.
4
Persistence strategy ambiguity
MongoDB is the runtime database, but SQL relational migration semantics are used throughout — no explicit commitment to either.
5
God-module concentration
Unconstrained feature accumulation in one controller compounds every other problem and blocks every high-priority fix.

13 / 14
final verdict

Conclusion

The system has the right intent but not the right implementation. Role separation is named but not enforced. Documentation is detailed but not accurate. The student flow is designed but not deployable.
most critical issue

Authorization is absent as a structural boundary — and the path to fixing it is currently broken. This must be resolved before any other security work is meaningful.

key strength

The architectural intent is sound. Role-based route files, enums, FormRequest scaffolding, and Inertia separation are good starting points. The work is refinement, not a rewrite.
